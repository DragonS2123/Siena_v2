# Siena v2 — Архитектурный документ

Статус: **черновик на утверждение (v1.2 — добавлена мульти-модельная делегация)**
Версия: 1.2 (MVP)
Платформа: Windows 11, Python 3.11+
Модель: Qwen (через Ollama, локально) — главная модель + делегируемые специализированные модели

---

## 1. Философия проекта

Siena v2 — это не оркестратор и не когнитивный пайплайн. Это тонкий Runtime вокруг
языковой модели, который не принимает ни одного содержательного решения сам.

| Роль          | Кто/что            | Ответственность                                      |
|---------------|--------------------|-------------------------------------------------------|
| Главный интеллект | Qwen, PRIMARY_MODEL (qwen3.5:9b) | Ведёт диалог, решает: ответить самой, вызвать инструмент, использовать память, делегировать задачу другой модели, отказаться |
| Специалист (по вызову) | Делегируемые модели (например CODE_MODEL — qwen2.5-coder:7b) | Выполняет узкую задачу, поставленную главной моделью, и никогда не отвечает пользователю напрямую |
| Тело          | Python Runtime     | Механически исполняет то, что попросила модель        |
| Органы чувств | Tools              | web_search, open_url, short_memory_*, long_memory_*, delegate_model |
| Рабочая память| Short-term Memory (JSON) | Временные факты текущей сессии, решает Qwen      |
| Внешний банк  | Long-term Memory (SQLite) | Устойчивые факты между сессиями, решает Qwen     |

Жёсткое правило, пронизывающее всю архитектуру:

> **Runtime не решает, ЧТО делать. Runtime решает только КАК это исполнить.**

Это правило теперь явно распространяется и на память: Runtime не решает, что важно,
что временно, а что нужно сохранить навсегда. Это решает только Qwen — через вызов
конкретного tool. С v1.2 то же самое верно и для выбора модели: Runtime не решает,
какая задача требует специализированной модели — это решает только главная модель
через вызов `delegate_model`, а не через `if`-условия в коде (раздел 12).

контракт `Tool`. Он импортирует оба стора (`short_memory_store`, `long_memory_store`),
но сам не содержит логики принятия решений — только вызов store-методов и упаковку
результата в `ToolResult`.

## 3. Поток выполнения (Agent Loop)

```
User
 │
 ▼
Session.append(user_message)
 │
 ▼
┌─────────────────────────────────────────────┐
│  LOOP (до MAX_ITERATIONS — см. раздел 7.4)   │
│                                               │
│   Qwen.chat(messages, tools=ALL_TOOL_SCHEMAS)│
│                                               │
│   ├── response содержит tool_calls?          │
│   │      │                                   │
│   │      ▼ да                                │
│   │   для каждого tool_call:                 │
│   │      Runtime.dispatch(name, args)        │
│   │      Session.append(tool_result)         │
│   │   → следующая итерация LOOP              │
│   │                                           │
│   └── response — финальный текст?            │
│          │                                   │
│          ▼ да                                │
│       Session.append(assistant_message)      │
│       return response.content                │
└─────────────────────────────────────────────┘
 │
 ▼
Final Answer → User
```

`ALL_TOOL_SCHEMAS` в v1.1 включает 8 инструментов: `web_search`, `open_url`,
`short_memory_save`, `short_memory_search`, `short_memory_clear`,
`long_memory_save`, `long_memory_search`, `long_memory_list`.

Ключевое: Python **никогда** не смотрит на `user_message` и не решает "надо ли
искать" или "надо ли запомнить". Он передаёт модели полный список инструментов
на каждой итерации — решение о вызове любого из них принимает исключительно Qwen.
Runtime не "подмешивает" память в промпт сам — единственный канал попадания
памяти к модели — явный `tool_call` от самой модели.

## 4. Формат Tool Calls

Стандартный формат Ollama/OpenAI function calling (совместимость с любой моделью,
поддерживающей tool calling).

### 4.1 Схемы инструментов памяти

```json
{
  "type": "function",
  "function": {
    "name": "short_memory_save",
    "description": "Сохранить временный рабочий факт/заметку/решение в кратковременную память текущей сессии.",
    "parameters": {
      "type": "object",
      "properties": {
        "text": { "type": "string", "description": "Текст, который нужно запомнить" }
      },
      "required": ["text"]
    }
  }
}
```

```json
{
  "type": "function",
  "function": {
    "name": "short_memory_search",
    "description": "Найти ранее сохранённые временные факты текущей сессии по запросу.",
    "parameters": {
      "type": "object",
      "properties": {
        "query": { "type": "string", "description": "Поисковый запрос" }
      },
      "required": ["query"]
    }
  }
}
```

```json
{
  "type": "function",
  "function": {
    "name": "short_memory_clear",
    "description": "Полностью очистить кратковременную память текущей сессии.",
    "parameters": { "type": "object", "properties": {} }
  }
}
```

```json
{
  "type": "function",
  "function": {
    "name": "long_memory_save",
    "description": "Сохранить факт в долговременную память НАВСЕГДА. Вызывать только когда пользователь явно просит что-то запомнить/сохранить/оставить в памяти надолго.",
    "parameters": {
      "type": "object",
      "properties": {
        "text": { "type": "string", "description": "Текст факта для сохранения" },
        "category": { "type": "string", "description": "Категория факта (например: architecture, preference, decision). Определяется моделью." },
        "importance": { "type": "string", "description": "Важность факта (например: low, medium, high). Определяется моделью." }
      },
      "required": ["text"]
    }
  }
}
```

```json
{
  "type": "function",
  "function": {
    "name": "long_memory_search",
    "description": "Найти факты в долговременной памяти по тексту или категории.",
    "parameters": {
      "type": "object",
      "properties": {
        "query": { "type": "string", "description": "Поисковый запрос (ищется по тексту и категории)" }
      },
      "required": ["query"]
    }
  }
}
```

```json
{
  "type": "function",
  "function": {
    "name": "long_memory_list",
    "description": "Показать последние сохранённые факты долговременной памяти.",
    "parameters": {
      "type": "object",
      "properties": {
        "limit": { "type": "integer", "description": "Сколько последних записей вернуть (по умолчанию 20)" }
      },
      "required": []
    }
  }
}
```

`web_search(query)` и `open_url(url)` — без изменений относительно v1.0.

### 4.2 Схема инструмента delegate_model (v1.2)

```json
{
  "type": "function",
  "function": {
    "name": "delegate_model",
    "description": "Делегировать узкую специализированную задачу другой модели (например, генерацию/анализ/рефакторинг кода) и получить её ответ как материал для собственного финального ответа. Ты никогда не пересылаешь результат этого вызова пользователю напрямую — сама проверяешь его и формируешь окончательный ответ.",
    "parameters": {
      "type": "object",
      "properties": {
        "model": { "type": "string", "description": "Имя модели-исполнителя, например qwen2.5-coder:7b" },
        "task": { "type": "string", "description": "Полная самодостаточная формулировка задачи для модели-исполнителя" }
      },
      "required": ["model", "task"]
    }
  }
}
```

Пример вызова:

```
delegate_model(
    model="qwen2.5-coder:7b",
    task="Напиши класс Repository на C# для EF Core."
)
```

### 4.3 Вызов и результат

Формат вызова модели и возврата результата не меняется относительно v1.0
(см. `role: assistant / tool_calls`, `role: tool / tool_call_id / content`).
Содержимое `content` для памяти — сериализованный JSON со списком найденных
записей или подтверждением сохранения, например:

```json
{"ok": true, "saved_id": 17, "text": "...", "category": "architecture", "importance": "high"}
```

### 4.4 Контракт инструмента внутри Python

Без изменений: `Tool.run(**kwargs) -> ToolResult`. Runtime валидирует только
**наличие и тип** обязательных полей (`text` — строка, `limit` — целое число
и т.п.) — это техническая валидация, а не смысловая оценка. Runtime никогда
не проверяет, "разумна" ли категория, "верна" ли важность, "стоит ли" вообще
сохранять — эти проверки не существуют в принципе.

## 5. Память как два независимых слоя

Ключевое архитектурное решение v1.1: память — это **не единая подсистема**,
а два независимых, не связанных друг с другом хранилища. Они не синхронизируются
автоматически, не имеют общего API и не имеют общей "логики важности". Единственное,
что их объединяет — оба доступны модели только через явные tool calls.

### 5.1 Short-term Memory (кратковременная, JSON)

- Файл: `memory/short_memory.json`.
- Формат — простой список записей: `{"id": 1, "timestamp": "...", "text": "..."}`.
- Назначение: рабочая память **текущей сессии** — временные факты, промежуточные
  решения, текущие задачи, заметки "для себя", которые Qwen считает полезными,
  чтобы не потерять нить разговора.
- JSON выбран намеренно (а не SQLite) — чтобы разработчик мог открыть файл
  глазами и проверить, **что именно и как** Siena решила сохранить. Это
  инструмент прозрачности и отладки качества решений модели, а не оптимизации.
- Живёт до explicit `short_memory_clear()` (вызывает только модель) или до
  ручной очистки разработчиком. Runtime сам, автоматически, между сессиями
  файл не чистит и не архивирует — это не его решение.
- Поиск v1: наивный keyword/substring-матчинг по `text`, без embeddings.

### 5.2 Long-term Memory (долговременная, SQLite)

- Файл: `memory/long_memory.sqlite3`.
- Назначение: то, что должно **переживать перезапуск программы и новые сессии** —
  устойчивые факты, решения проекта, предпочтения пользователя, важные
  архитектурные выводы.
- Схема:

```sql
CREATE TABLE IF NOT EXISTS long_memory (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    text TEXT NOT NULL,
    category TEXT,
    importance TEXT,
    source TEXT,
    metadata_json TEXT
);

CREATE INDEX IF NOT EXISTS idx_long_memory_text ON long_memory(text);
CREATE INDEX IF NOT EXISTS idx_long_memory_category ON long_memory(category);
CREATE INDEX IF NOT EXISTS idx_long_memory_importance ON long_memory(importance);
CREATE INDEX IF NOT EXISTS idx_long_memory_created_at ON long_memory(created_at);
```

- Поля `category` и `importance` заполняются **моделью** через аргументы tool
  call — Runtime их не валидирует по смыслу (нет enum, нет справочника). Модель
  вольна использовать любую формулировку.
- Поле `source` и `metadata_json` — технические, заполняются Runtime
  автоматически (например, `source="siena_v2"`, `metadata_json` с session id/
  timestamp контекстом). Это не смысловое решение, а служебная метка, аналог
  timestamp — модель их не задаёт и не может задать через tool call.
- Поиск v1 — простой, без embeddings/RAG:
  - `LIKE '%query%'` по `text`;
  - `LIKE '%query%'` по `category`;
  - сортировка `ORDER BY created_at DESC`;
  - `LIMIT` (по умолчанию 20 для `long_memory_list`, без ограничения по
    умолчанию для `long_memory_search`, кроме разумного технического лимита).

### 5.3 Правило вызова long_memory_save (поведенческое, задаётся системным промптом)

Long-term memory — это **дорогое, необратимое** (в бытовом смысле — переживает
сессии) действие, поэтому модель должна вызывать `long_memory_save` **только**
когда пользователь явно попросил что-то запомнить/сохранить/оставить в памяти/
перенести в долгую память или дал явную аналогичную команду. Это поведенческое
правило, которое нельзя закодировать в Python (это было бы смысловым решением
Runtime за модель) — оно закрепляется исключительно в системном промпте
(раздел 11) и является поведенческим контрактом с моделью, а не программным
ограничением.

Пример:

Пользователь: *"Сохрани: Siena_v2 должна оставаться Python-first архитектурой."*

Qwen вызывает:
```
long_memory_save(
    text="Siena_v2 должна оставаться Python-first архитектурой.",
    category="architecture",
    importance="high"
)
```

После результата tool call модель отвечает пользователю обычным текстом,
подтверждая сохранение (текст подтверждения формулирует сама модель).

### 5.4 Что Runtime не делает в отношении памяти (явный список)

Runtime **может**:
- валидировать аргументы tools (наличие, тип);
- выполнять SQL/JSON операции чтения и записи;
- возвращать `ToolResult`;
- логировать каждый вызов;
- обрабатывать инфраструктурные ошибки (повреждённый файл/база, блокировка);
- останавливать loop по `MAX_ITERATIONS`.

Runtime **не может**:
- сам решать, что сохранить;
- сам вызывать `short_memory_save`/`long_memory_save`;
- сам искать в памяти без запроса модели;
- сам выбирать категорию или важность;
- сам оценивать, "достаточно ли явно" пользователь попросил сохранить (это
  решает модель, читая системный промпт и сообщение пользователя);
- сам подмешивать содержимое short/long memory в промпт в обход tool call.

## 6. Взаимодействие компонентов

```
main.py
  → создаёт Session, Logger, ToolRegistry (авто-сбор всех tools/*)
  → читает ввод пользователя в цикле REPL
  → вызывает agent_loop.run(user_input, session)

agent_loop.py
  → ollama_client.chat(session.messages, tools=registry.schemas())
  → если tool_calls: registry.dispatch(name, args) для каждого
  → логирует каждый шаг через logging_.logger
  → возвращает финальный ответ в main.py

ollama_client.py
  → HTTP-обёртка над локальным Ollama (http://localhost:11434/api/chat)
  → не содержит никакой бизнес-логики, только транспорт + сериализация
  → chat() с v1.2 принимает необязательный override модели на конкретный вызов
    (используется только tools/delegate_model.py; agent_loop.py всегда вызывает
    chat() без override — он говорит только с PRIMARY_MODEL)

tools/memory_tools.py
  → тонкие обёртки над memory/short_memory_store.py и memory/long_memory_store.py
  → short_memory_save/search/clear и long_memory_save/search/list —
    единственные точки входа модели в оба хранилища

tools/delegate_model.py
  → тонкая обёртка: вызывает ollama_client.chat() с указанной моделью и
    изолированным одноразовым списком messages=[{"role": "user", "content": task}]
    (не историей основной сессии — это разовая задача, а не продолжение диалога)
  → результат — просто text-ответ делегируемой модели, упакованный в ToolResult
  → сама НЕ решает, какую модель вызывать — имя модели приходит в аргументе tool call

memory/short_memory_store.py
  → CRUD + наивный поиск по memory/short_memory.json

memory/long_memory_store.py
  → sqlite3-соединение с memory/long_memory.sqlite3, CRUD + LIKE-поиск
  → создаёт схему (CREATE TABLE IF NOT EXISTS ...) при первом запуске
```

## 7. Обработка ошибок

### 7.1 Ошибки исполнения инструмента (модель должна знать о них)

Примеры: сайт недоступен, поиск не дал результатов, short-memory файл пуст,
long-memory запрос не нашёл совпадений, делегируемая модель вернула ошибку/
недоступна/не отвечает в срок.

→ Оборачиваются в `ToolResult(ok=False, error=...)` → отправляются модели как
`role: tool`. Модель сама решает дальнейшее действие (например, попробовать
другую формулировку задачи, или ответить самостоятельно без делегирования).

`delegate_model` — отдельный случай в этой категории: запрос **неизвестного**
имени модели (например, опечатка или модель, которой нет в `config.DELEGATE_MODELS`)
— это тоже `SienaToolError`, а не инфраструктурный сбой: Runtime технически
знает список зарегистрированных для делегирования моделей (раздел 12), и
проверка "есть ли такая модель в списке" — валидация присутствия/формата
аргумента (раздел 4.4), а не смысловая оценка "стоило ли её вызывать".

### 7.2 Ошибки формата вызова (модель дала невалидные аргументы)

Например, отсутствует обязательное поле `text` в `long_memory_save`, или
`limit` — не число.

→ Возвращаются модели как `tool` result с `ok=False` и описанием проблемы,
чтобы модель могла скорректировать вызов сама.

### 7.3 Инфраструктурные сбои (модели нет кого спрашивать)

Примеры: Ollama не запущен, повреждён `short_memory.json`, `long_memory.sqlite3`
заблокирован другим процессом или повреждён, диск переполнен.

→ Логируются как ERROR с полным traceback, эскалируются пользователю напрямую
без обращения к модели. Runtime не пытается "тихо починить" данные (например,
не пересоздаёт повреждённый `short_memory.json` пустым файлом без явного лога
и решения оператора).

### 7.4 Защита цикла (MAX_ITERATIONS)

Без изменений относительно v1.0 — инженерная защита от зацикливания/затрат,
не связанная с содержанием вызовов (в том числе к памяти).

---

## 8. Логирование

Без структурных изменений относительно v1.0. Дополнительно логируются события
памяти как отдельные `event` в JSONL:

```json
{"ts": "...", "event": "tool_dispatch", "name": "long_memory_save", "args": {"text": "...", "category": "architecture", "importance": "high"}}
{"ts": "...", "event": "tool_result", "name": "long_memory_save", "ok": true, "content": "{\"saved_id\": 17}"}
```

Это даёт полную трассировку **каждого** решения модели о памяти — что именно
и когда она посчитала нужным сохранить, найти или очистить.

С v1.2 аналогично логируется каждое делегирование задачи другой модели:

```json
{"ts": "...", "event": "model_delegate", "from": "qwen3.5:9b", "to": "qwen2.5-coder:7b", "task": "Напиши класс Repository..."}
{"ts": "...", "event": "model_delegate_result", "model": "qwen2.5-coder:7b", "duration_ms": 4231, "tokens": 512, "ok": true}
```

Это даёт трассировку того, **какие задачи и какой специализированной модели**
Siena решила поручить — без этого нельзя было бы отличить "модель ответила
сама" от "модель делегировала и переформулировала чужой ответ".

---

## 9. Расширяемость для будущих инструментов

Без изменений относительно v1.0. Дополнительно: замена наивного поиска
(`LIKE` для long-term, substring для short-term) на векторный/эмбеддинг-поиск
в будущем — это замена **внутренней реализации** `long_memory_store`/
`short_memory_store`, без изменения контракта tools (`long_memory_search`,
`short_memory_search` остаются тем же именем и той же сигнатурой для модели).

С v1.2: любая будущая специализированная модель (vision, speech, embedding,
reasoning и т.д.) добавляется **только** как ещё одна запись в
`config.DELEGATE_MODELS` — она уже вызываема через существующий `delegate_model`
tool. `core/agent_loop.py` при этом не меняется вообще: он ничего не знает про
конкретные модели, только про универсальный tool-контракт. Отдельный
`vision_delegate`/`speech_delegate` tool не нужен, пока задача укладывается в
формат "модель, задача, текстовый ответ" — см. раздел 12.

---

## 10. Что НЕ входит в v1

LoRA, RAG, embeddings, Reflection, Observation, PersonalityGrowth, TrainingCase,
Auto Learning, Dataset Export, Cognitive Engine, Research Decision Engine —
отсутствуют полностью. Это в равной степени относится и к памяти: ни short-term,
ни long-term хранилище в v1 не получают семантический/векторный поиск — только
keyword/LIKE.

## 11. Системный промпт (явный поведенческий контракт с моделью)

Системный промпт — единственное место, где закрепляются поведенческие правила
использования инструментов (включая правило из раздела 5.3). Он не заменяет
код, а дополняет tool-схемы объяснением **когда** модель должна их применять:

```
Ты — Siena.
Ты сама решаешь, когда использовать инструменты.
Runtime не будет искать информацию, сохранять память или выбирать источник за тебя.

Если тебе нужна актуальная информация — вызови web_search.
Если тебе нужно вспомнить временные данные текущей работы — вызови short_memory_search.
Если в ходе текущей сессии появился полезный рабочий факт — можешь вызвать short_memory_save.
Если пользователь явно просит что-то сохранить надолго — вызови long_memory_save.
Не сохраняй в long-term memory без явной просьбы пользователя.
Не проси Runtime сделать выбор за тебя.
```

Это единственный текст в системе, который управляет "осторожностью" вызова
`long_memory_save` — Python не содержит и не должен содержать никакой проверки
"явно ли попросил пользователь".

## 12. Мульти-модельная делегация (v1.2)

Главный принцип не меняется: Runtime не анализирует запрос пользователя, не
выбирает модель и не решает, когда задача "достаточно специализирована".
Единственная, кто это решает — главная модель (`PRIMARY_MODEL`), через явный
вызов `delegate_model`.

### 12.1 Роли моделей

| Модель | Config-константа | Роль |
|---|---|---|
| `qwen3.5:9b` | `PRIMARY_MODEL` | Главный интеллект. Ведёт диалог, управляет tools и памятью, принимает все смысловые решения. Всегда формирует финальный ответ пользователю. |
| `qwen2.5-coder:7b` | `CODE_MODEL` | Специалист по коду: генерация, исправление, анализ, рефакторинг, объяснение кода. Никогда не отвечает пользователю напрямую. |

Главной моделью всегда остаётся `PRIMARY_MODEL` — именно она ведёт `Session`,
именно её вызывает `core/agent_loop.py` на каждой итерации. Делегируемые модели
не участвуют в основном цикле и не видят историю диалога — они получают только
текст конкретной задачи (раздел 12.3).

### 12.2 Tool: delegate_model(model, task)

Схема — раздел 4.2. Поток выполнения:

```
PRIMARY_MODEL решает делегировать
 │
 ▼
tool_call: delegate_model(model="qwen2.5-coder:7b", task="...")
 │
 ▼
Runtime: registry.dispatch("delegate_model", args)
 │
 ▼
tools/delegate_model.py:
  - проверяет, что model входит в config.DELEGATE_MODELS (техническая
    валидация допустимого значения, не смысловая оценка задачи)
  - вызывает ollama_client.chat(messages=[{"role":"user","content":task}],
    model=<запрошенная модель>) — новый, изолированный от Session разговор
 │
 ▼
ToolResult(ok=True, content=<текст ответа delegate-модели>)
 │
 ▼
Session.add_tool_result(...) → возвращается PRIMARY_MODEL как role:tool
 │
 ▼
PRIMARY_MODEL сама проверяет результат и формирует финальный ответ пользователю
```

Delegate-модель **никогда** не отвечает пользователю напрямую — её вывод это
просто содержимое `tool` message, один из входов для следующего шага
`PRIMARY_MODEL`, ничем принципиально не отличающийся от результата `web_search`
или `long_memory_search`.

### 12.3 Что Runtime делает и чего не делает

Runtime **может**:
- принять `tool_call` вида `delegate_model(model, task)`;
- проверить, что `model` есть в списке `config.DELEGATE_MODELS` (известная,
  сконфигурированная модель) — если нет, вернуть `SienaToolError` (раздел 7.1);
- вызвать указанную модель через `ollama_client.chat()` с одноразовым
  сообщением `task`;
- вернуть её текстовый ответ как `ToolResult` главной модели;
- залогировать `model_delegate` / `model_delegate_result` (раздел 8).

Runtime **не может**:
- решать, что задача "похожа на код" и сама вызывать `delegate_model`
  — никаких условий вида `if "код" in user_input`, `if programming`,
  `if python` в коде быть не должно; выбор — исключительно за `PRIMARY_MODEL`;
  Runtime реагирует только на явный `tool_call`, как и для любого другого tool;
- отправлять ответ delegate-модели пользователю напрямую, минуя `PRIMARY_MODEL`;
- решать, какая delegate-модель "лучше подходит" — это не Runtime выбирает
  между `CODE_MODEL` и, скажем, будущей `VISION_MODEL`, а сама главная модель,
  передавая конкретное имя в аргументе `model`.

### 12.4 Config

```python
PRIMARY_MODEL = "qwen3.5:9b"
CODE_MODEL = "qwen2.5-coder:7b"

# Реестр моделей, которые разрешено вызывать через delegate_model.
# Runtime использует этот список только для технической проверки "модель
# существует и сконфигурирована", а не для выбора, какую вызывать.
DELEGATE_MODELS = {
    CODE_MODEL: "Специализированная модель программирования (код, рефакторинг, анализ, объяснение кода).",
}
```

Добавление будущей модели (vision/speech/embedding/reasoning) — это всего
лишь новая запись в `DELEGATE_MODELS` (раздел 9); `core/agent_loop.py` не
меняется.

### 12.5 System Prompt (дополнение к разделу 11)

```
Если задача требует:
- написания программного кода;
- исправления программного кода;
- анализа кода;
- рефакторинга;
- объяснения большого блока кода;

ты МОЖЕШЬ вызвать delegate_model() и поручить задачу модели qwen2.5-coder:7b.

После получения результата сама проверь его и сформируй окончательный ответ пользователю.
Никогда не отдавай пользователю ответ модели напрямую.
Всегда оставайся главным интеллектом Siena.
```
