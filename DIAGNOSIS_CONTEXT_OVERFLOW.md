# Диагностика: Siena отвечает пустой строкой (done_reason="length")

Статус: **расследование завершено, причина подтверждена кодом и логами**.
Исправления **не вносились** — по требованию задачи это только отчёт.

---

## TL;DR

Session (история диалога) в Runtime **никогда не обрезается** — растёт бесконечно
и передаётся Ollama **целиком** на каждом вызове. Модель `qwen3.5:9b` физически
поддерживает контекст 262144 токена, но Ollama, если ей не передать `num_ctx`
явно, загружает её с окном **4096 токенов** (подтверждено `curl /api/ps` —
`"context_length": 4096`, см. раздел 9). Единственная точка, где сейчас
конструируется вызов модели (`core/ollama_client.py`), **вообще не передаёт
`options`** — ни `num_ctx`, ни что-либо ещё.

Как только суммарный prompt (system + вся история + все результаты tools,
включая полные тексты `web_search`/`open_url`) приближается к 4096 токенам,
Ollama'е физически некуда генерировать ответ — она возвращает `done_reason:
"length"` и `content: ""` почти сразу. Дальше эффект самоусиливается: Session
продолжает расти, prompt только приближается к потолку ещё плотнее — отсюда
"повторяется почти на каждом запросе".

Отдельно и по факту хуже: в `api/server.py` (backend, который реально
обслуживает UI) `Session` — **глобальный объект уровня процесса**, создаётся
один раз при старте сервера и никогда не пересоздаётся между разговорами. Она
растёт с момента запуска backend'а и до его перезапуска — не "на сессию", а
"навсегда, пока сервер жив".

## 3. Параметры вызова Ollama — что реально передаётся

Смотрите `core/ollama_client.py::chat()` (текущая реализация):

```python
response = self._client.chat(
    model=target_model,
    messages=messages,
    tools=tools or None,
    think=self._think,
)
```

Передаются только `model`, `messages`, `tools`, `think`. **Параметр `options`
не передаётся вообще.** Ни `num_ctx`, ни `num_predict`, ни `temperature`,
ни `top_p`, ни `repeat_penalty`, ни `stop`, ни `seed` — нигде в коде проекта
(`grep -rn "num_ctx\|num_predict\|options=" *.py core/ tools/ api/` — пусто).

Что использует Ollama по умолчанию, когда `options` не передан (проверено
эмпирически, не по документации):

| Параметр | Источник дефолта | Фактическое значение (проверено) |
|---|---|---|
| `num_ctx` | Ollama runtime default (нет `OLLAMA_CONTEXT_LENGTH` в env, нет override в коде) | **4096** — подтверждено `curl http://localhost:11434/api/ps` → `"context_length": 4096` для загруженной `qwen3.5:9b` |
| `temperature`, `top_k`, `top_p`, `presence_penalty` | из Modelfile модели (`ollama show qwen3.5:9b --parameters`) | `temperature=1`, `top_k=20`, `top_p=0.95`, `presence_penalty=1.5` — заданы в самой модели, не Runtime'ом |
| `num_predict` | Ollama default | не переопределяется нигде; по факту генерация не режется отдельным лимитом токенов — во всех 8 случаях `done_reason="length"` останов происходит из-за нехватки места в контексте (см. `eval_count` от 4 до 51 — то есть модель успевала сгенерировать буквально несколько токенов, а не "много и обрезали") |
| `stop`, `seed` | Ollama default | не заданы нигде |

**Вывод по п.3: единственный технически значимый параметр здесь — `num_ctx`,
и он НЕ управляется Runtime'ом. Используется дефолт Ollama (4096), который в
64 раза меньше нативного предела модели (262144).**

## 4. Передаётся вся история или только последние сообщения

**Вся история, без исключений.** `core/agent_loop.py:30`:

```python
raw_response = ollama_client.chat(session.get_messages(), tools=registry.schemas())
```

`session.get_messages()` (`core/session.py:23-24`) возвращает **весь**
внутренний список `self.messages`, который только растёт (`add_user`,
`add_assistant_raw`, `add_tool_result` — все три только `.append(...)`,
ни одного места, где что-то удаляется или обрезается — см. `core/session.py`
целиком, 25 строк, других методов нет).

Пример реальной структуры (из логов, сокращённо, к 178-му сообщению):

```
System (system prompt, см. config.SYSTEM_PROMPT)
User 1
Assistant 1 (или tool_calls)
[Tool result 1..N — если были вызовы]
Assistant 1'
User 2
Assistant 2
...
User 48
Assistant ~95-е по счёту сообщение
```

**Вывод по п.4: подтверждено — передаётся вся история, не "последние N
сообщений". Окна/лимита сообщений не существует нигде в коде.**

## 5. Agent Loop — второй и последующие итерации после ToolResult

Тот же вызов `ollama_client.chat(session.get_messages(), ...)` выполняется на
**каждой** итерации цикла `for iteration in range(1, max_iterations + 1)`
(`core/agent_loop.py:29-30`) — без каких-либо изменений между итерациями.
Внутри одного пользовательского запроса, если модель вызвала несколько
инструментов подряд (несколько итераций до финального ответа), **на каждую
итерацию заново отправляется вся история**, включая уже добавленные
`tool_result`-сообщения из предыдущих итераций этого же запроса. Отдельного
"облегчённого" контекста для повторных итераций не существует.

Это объясняет ещё один паттерн из логов: просадка `prompt_eval_count` до 2050
на строке `2026-07-05T11:34:04` (запись #57) сразу **после** нескольких
подряд `done_reason="length"` — это не исправление со стороны Runtime (в коде
нет такого места), а, по всей видимости, собственное поведение Ollama: когда
запрос не помещается в `num_ctx`, сервер молча обрезает/переоценивает более
старую часть контекста при следующей загрузке, а не отклоняет запрос ошибкой.
Т.е. модель в какой-то момент могла перестать "видеть" часть истории (включая,
возможно, часть system prompt) без единого сообщения об этом в логах Runtime —
самостоятельно наблюдаемый, но не объяснённый нашим кодом эффект.

**Вывод по п.5: да, история передаётся целиком на каждой итерации внутри
одного запроса, без разницы между первой и последующими.**

## 7. Session — растёт ли бесконечно

**Да, подтверждено на двух уровнях.**

**(a) core/session.py** — сам класс `Session` не имеет ни одного метода
удаления/обрезки сообщений. Единственный способ его "очистить" —
пересоздать объект (`Session(system_prompt)`).

**(b) api/server.py (реальный backend для UI) — это отдельная, более
серьёзная находка:**

```python
session = Session(config.SYSTEM_PROMPT)   # строка 88, уровень модуля
...
@app.post("/api/chat")
async def chat(request: ChatRequest) -> dict[str, Any]:
    ...
    session.add_user(text)                # тот же объект каждый раз
```

`session` создаётся **один раз при импорте модуля** (то есть один раз за
время жизни backend-процесса) и используется как разделяемое состояние для
**всех** запросов `/api/chat`, сколько бы их ни пришло. Ни один эндпоинт не
создаёт новую `Session`, не сбрасывает её и не предлагает пользователю "новый
разговор" — проверено также со стороны UI:
`grep -rn "session" "Siena v2 Control Panel UI/src"` находит только
`sessionSecs` — таймер аптайма интерфейса, никак не связанный с backend'ом.

Это значит: с момента запуска `uvicorn api.server:app` (через
`start_backend.bat`) история растёт **непрерывно и без сброса**, пока сервер
не будет перезапущен. Для контекста: лог одного дня показывает, что реальный
процесс backend'а работал несколько часов (`08:54` → `11:35` по меткам в
логе) без перезапуска — и это ровно тот путь, который использует UI.

**Вывод по п.7: Session растёт бесконечно в обоих смыслах — (a) в коде класса
нет предела, (b) в реальном backend-процессе это ещё и общий для всех
диалогов синглтон, который не сбрасывается никогда.**

## 8. Memory — не подмешивается ли автоматически в prompt

**Не подтвердилось — память не самовольно подмешивается.** Проверено:

- `core/agent_loop.py` и `core/session.py` не импортируют и не обращаются
  ни к `ShortMemoryStore`, ни к `LongMemoryStore` напрямую — единственный
  канал попадания памяти в историю — через `tools/memory_tools.py`,
  вызываемый **только** по explicit `tool_call` от модели
  (`tools/registry.py::dispatch`, вызывается из `agent_loop.py:85` только
  в ответ на `tool_calls`, пришедшие в ответе модели).
- Ни в `main.py`, ни в `api/server.py` нет кода, который бы читал
  `short_memory.json`/`long_memory.sqlite3` и добавлял их содержимое в
  `Session` в обход tool call.

**Однако** результаты `short_memory_search`/`long_memory_search`/`_save`,
однажды попав в историю через explicit tool call, **тоже остаются там
навсегда**, как и любой другой tool result (см. п.6) — то есть память не
"подмешивается автоматически", но результат её **однократного, вполне
легитимного** использования продолжает занимать место в контексте до конца
жизни Session, внося свой (небольшой, судя по размерам в п.6: 39-494 символа)
вклад в общий рост.

**Вывод по п.8: автоматической подмешки памяти нет — эта часть архитектуры
работает как задумано. Единственный вклад памяти в проблему — обычный
"эффект накопления", общий для всех сообщений истории.**

## 9. Максимальный размер контекста модели

```
$ ollama show qwen3.5:9b   →  model_info["qwen35.context_length"] = 262144
$ curl :11434/api/ps       →  "context_length": 4096   (реально загруженное окно)
```

Модель **нативно поддерживает 262144 токена**. Ollama загрузила её с окном
**4096** — потому что ни один вызов в коде проекта не передаёт `num_ctx`, а
переменная окружения `OLLAMA_CONTEXT_LENGTH` не установлена (проверено
`env | grep -i OLLAMA` — пусто). Это чистый дефолт самой Ollama, никак не
Siena-специфичный.

4096 — это **1.5%** от возможностей модели. Даже скромное увеличение (не
обязательно до максимума — полный 262144 контекст резко увеличит потребление
VRAM/RAM и время обработки промпта) сняло бы проблему на порядок дольше, чем
сейчас. Конкретное число — предмет отдельного решения с учётом доступной
памяти видеокарты/RAM; в разделе 10 это оставлено как открытый технический
параметр, а не догма.

**Вывод по п.9: используется 4096 из 262144 доступных — с большим запасом
можно поднять этот потолок технически, не трогая философию проекта.**

## 10. Решение, не нарушающее философию Siena_v2

Формулировка задачи явно разделяет два типа решений:

- **Смысловые решения** (что важно, что удалить, что суммировать) —
  однозначно **не Runtime**, это была бы прямая узурпация роли модели.
- **Техническое управление контекстом** (посчитать токены, не отправить то,
  что не помещается, обрезать по границе, хранить полную историю отдельно) —
  явно разрешено самой постановкой задачи, и это именно то, чего сейчас не
  хватает.

Направления решения (без реализации, только описание — исполнение будет
отдельным шагом после этого отчёта):

1. **Явно задать `num_ctx`** в `options` при вызове `ollama.Client.chat()` —
   технический параметр транспорта (как `timeout`), не смысловое решение.
   Подняло бы потолок (например, до 16384/32768 — конкретное число зависит от
   доступной VRAM, это открытый вопрос) и отодвинуло бы момент проблемы, но
   **не устранило бы её**, так как Session всё равно растёт бесконечно —
   это отсрочка, а не решение.
2. **Runtime считает токены** (например, через `prompt_eval_count`, который
   Ollama и так возвращает в каждом ответе, или через локальный токенайзер)
   и **технически** не отправляет сообщения, которые не помещаются в
   `num_ctx` — это не решение "что важно", а решение "что физически влезает",
   ровно как формулирует сама задача.
3. **Хранить полную историю отдельно** (например, всю Session — на диск/в
   памяти без изменений, как сейчас) **и передавать модели только окно,
   технически умещающееся в контекст** — например, срез последних N
   сообщений по счётчику токенов, а не по смысловой важности. Это
   **тримминг по границе, а не суммаризация** — Runtime не решает, что
   "неважно", он просто не передаёт то, что физически не помещается,
   технически измеримым способом (токены), а не оценочным.
4. **Отдельно (независимо от лимита контекста)**: `api/server.py` должен
   создавать новую `Session` на разговор/соединение, а не держать один
   глобальный `Session` на весь процесс — иначе даже с увеличенным `num_ctx`
   и тримингом граница просто отодвинется по времени, но актуальность вопроса
   "чья это история" (несколько вкладок UI, несколько пользователей) всё
   равно не решена архитектурно. Это тоже техническое, не смысловое решение
   (время жизни объекта, не его содержимое).

Все четыре пункта укладываются в явно разрешённый список задачи:
"считать токены; не отправлять сообщения, которые уже не помещаются; удалять
самые старые сообщения; хранить полную историю отдельно; передавать модели
только технически допустимый объём контекста" — ни один не требует от
Runtime решать, что "важно", "актуально" или "стоит забыть".

## Итоговая причинно-следственная цепочка

1. `api/server.py` держит один `Session` на весь процесс, без сброса
   (раздел 7б) →
2. `Session` не имеет механизма удаления сообщений (раздел 7а) →
3. Каждый tool call (особенно `web_search`/`open_url`) добавляет в историю
   результаты по 1-2 тыс. символов навсегда (раздел 6) →
4. `core/agent_loop.py` передаёт **всю** историю на **каждый** вызов модели,
   включая повторные итерации внутри одного запроса (разделы 4-5) →
5. `core/ollama_client.py` не передаёт `options.num_ctx`, поэтому Ollama
   грузит модель с дефолтным окном 4096 токенов вместо нативных 262144
   (разделы 3, 9) →
6. `prompt_eval_count` монотонно растёт (раздел 2) и рано или поздно
   пересекает 4096 →
7. как только prompt почти/полностью заполняет `num_ctx`, свободного места
   под генерацию не остаётся → Ollama возвращает `done_reason: "length"`
   почти немедленно (`eval_count` падает до единиц токенов — 4, 17, 18, 20,
   37, 51 в зафиксированных случаях) → `content` пустой, потому что модель
   физически не успела ничего сгенерировать до обрезки →
8. поскольку Session после этого не уменьшается, следующий запрос стартует
   с ещё большим prompt'ом — отсюда "повторяется почти на каждом запросе".

## Файлы, отвечающие за проблему

| Файл | Роль в проблеме |
|---|---|
| `api/server.py:88` | Глобальный `Session` на весь процесс без сброса — корневой источник неограниченного роста в реальном UI-сценарии |
| `core/session.py` | Класс `Session` без единого механизма удаления/лимита сообщений |
| `core/agent_loop.py:30` | Передаёт `session.get_messages()` (всю историю) на каждый вызов и каждую итерацию цикла |
| `core/ollama_client.py:22-49` | Не передаёт `options` (в частности `num_ctx`) вообще — Ollama использует дефолт 4096 |
| `tools/web_search.py`, `tools/open_url.py` | Кладут в историю крупные (1-2 тыс. символов) результаты, которые никогда не уменьшаются/не удаляются |
